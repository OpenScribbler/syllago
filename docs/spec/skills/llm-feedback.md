# Gemini

This is a robust, well-reasoned specification. You’ve clearly learned from the "wild west" of early package manager security (like NPM and PyPI) by baking in integrity and identity from the start.

However, since you asked me to "tear it to shreds," here is a critique focusing on technical edge cases, potential "footguns" for implementers, and security gaps.

---

## 1. High-Level Architecture & Strategy

### The "Sidecar Deletion" Weakness (Sec 11.4)
You acknowledge this, but it’s a critical UX/Security failure point. If an agent loads a `skill` from a directory where `meta.yaml` was deleted, does it default to "Unrusted"?
* **Critique:** If the agent is "helpful" and runs it anyway, the provenance is purely decorative.
* **Fix:** The spec should strongly recommend a **"Secure-by-Default"** mode where agents *refuse* to execute content lacking a valid `meta.yaml` unless a specific `--untrusted` flag is passed.

### Semantic Versioning vs. Integer Versioning (Sec 6.3.4)
You use a simple integer `version`. While clean, it fails to communicate "Breaking Changes" to an AI agent. 
* **Critique:** If a `skill` updates from version 3 to 4 and changes its input schema, an agent using it might break.
* **Fix:** Consider SemVer (e.g., `1.2.3`). If an agent is autonomous, it needs to know if an update is a "safe" patch or a breaking refactor.

---

## 2. Technical Inconsistencies & "Papercuts"

### The YAML vs. JSON Canonicalization Paradox (Sec 8)
You use `meta.yaml` for storage but require **JCS (RFC8785)** for hashing.
* **Critique:** YAML is notoriously difficult to canonicalize because of its flexibility (anchors, aliases, different string styles). Converting YAML to JSON just to hash it introduces a massive risk: if two different YAML parsers interpret a corner case differently (e.g., a floating-point number or a timestamp), they will generate different JSON, and the `meta_hash` will fail.
* **Fix:** Explicitly state that the `meta.yaml` must be parsed into an internal data structure and then serialized to JSON *strictly* following RFC8785. Warn developers that "re-encoding" the YAML string itself will not work.

### The "Empty File" and "Zero Content" Problem (Sec 7.5)
* **Critique:** You state "If enumeration produces zero files... MUST be rejected." But what if a directory contains only `.hidden` files that are ignored by some OS-level `ls` but captured by your glob? 
* **Fix:** Be explicit about whether `.DS_Store` (macOS) or `Thumbs.db` (Windows) should be globally excluded. If a user's `content_hash` changes just because they opened the folder in Finder, they will hate this spec.

### Symlink "In-Or-Out" Complexity (Sec 7.3)
* **Critique:** Resolving internal symlinks but excluding external ones is smart, but "External" is relative. If a user has a monorepo where a skill links to a shared `utils/` folder *outside* the skill directory but *inside* the repo, the hash fails.
* **Fix:** This forces users to copy-paste code instead of dry-linking. This might lead to "Provenance Fatigue" where users just stop using symlinks.

---

## 3. Security & Adversarial Attacks

### Signature Malleability (Sec 9.1)
Your signing input is `content_hash + \n + meta_hash`. 
* **Critique:** This is a "naked" signature. It doesn't include a **Context String** or **Domain Separator**.
* **The Risk:** If another protocol uses the same signing format for something else, a signature from that protocol could be replayed here.
* **Fix:** The signing input should be prefixed with a protocol string, e.g., `ACP-V1:content_hash\nmeta_hash`.

### The `generated_by` Spoofing (Sec 6.3.7)
* **Critique:** You label this as informational. However, in an enterprise setting, an admin might create a rule: "Only allow content generated by `internal-safe-agent/1.0`."
* **The Risk:** Since this field is part of the `meta_hash`, a human author can manually type `generated_by: claude-3.5-sonnet` into their hand-written malicious script. The hash and signature will be valid, but the claim is a lie.
* **Fix:** Add a warning that `generated_by` is **Attestation**, not **Verification**, unless the signing identity (Sigstore) matches a known AI-Agent service account.

### Lineage Squatting (Sec 10)
* **Critique:** I can create a malicious skill and claim it is `derived_from` a very popular, trusted skill. 
* **The Risk:** Users see the "Trusted" source in the metadata and lower their guard.
* **Fix:** Consumers MUST NOT display the `derived_from` source as "Verified" unless they can independently verify the `source_hash` against a known registry.

---

## 4. Logical Gaps in the Schema

### Missing: License Field
* **Critique:** For a "Provenance" spec, the lack of a `license` field is a glaring omission for legal/enterprise adoption.
* **Fix:** Add an optional `license` (SPDX format) field.

### Missing: Target Environment/Platform
* **Critique:** AI content is often platform-specific (e.g., a "Claude Code" hook won't work in "Cursor"). 
* **Fix:** The `type` field is a good start, but a `platform: [claude-code, cursor, windsurf]` field would prevent users from installing incompatible (and thus potentially broken/insecure) content.

---

## Summary of "The Shredding"

1.  **The JCS Requirement is the biggest implementation hurdle.** YAML-to-JCS-JSON is a friction point that will lead to "Hash Mismatch" GitHub issues.
2.  **The lack of Domain Separation in the signature** is a cryptographic "no-no."
3.  **The "Sidecar Deletion" scenario** needs a stronger "Fail-Closed" recommendation to be truly secure.

**Verdict:** This is 90% of the way to a production-grade standard. Fix the signature prefixing and the YAML/JSON parsing ambiguity, and you have something very solid.

**One final question to guide the next iteration:** How should the spec handle "Transitive Provenance"—if Skill A is derived from Skill B, which is derived from Skill C, should `meta.yaml` only store the immediate parent or the full chain?

## Security Issues

**Selective field stripping isn't addressed.** Section 11.4 covers sidecar removal, but an attacker can also just remove the `signature` field from `meta.yaml`. The content and meta hashes still verify. The spec says consumers "SHOULD distinguish unsigned content from signed content" — but that's it. There's no mechanism for a publisher to assert "this content MUST be signed" in a way consumers are required to enforce. A registry could strip signatures and downgrade everything to unsigned.

**The signing input is underspecified.** Section 9.1 says the signature covers "the `content_hash` value, a newline character (0x0A), and the `meta_hash` value." It doesn't state the encoding (UTF-8 is implied elsewhere but never stated here). More importantly, it doesn't define the exact byte representation — is it `sha256:` + 64 hex chars? Just the 64 hex chars? SHA-256 binary? This matters for interoperability. Two implementations could parse `content_hash` differently and produce different signing inputs.

**YAML type coercion is a ticking clock for `meta_hash`.** Section 8 says serialize the YAML fields to canonical JSON via JCS. But `published_at` in YAML is often auto-parsed as a `timestamp` type by YAML parsers, not a string. If Parser A returns a string and Parser B returns a datetime object, serialization to JSON may differ. You need to specify that all YAML scalar fields are treated as strings unless they're explicitly typed (integers, booleans). Failing to nail this down will produce hash mismatches across implementations and nobody will know why.

**No algorithm agility path.** Section 11.5 says "a new `meta_version` with an updated algorithm would be required" if SHA-256 is compromised. But `meta_version` bumps are currently fail-closed ("consumers MUST reject meta.yaml files with an unrecognized `meta_version`"). There's no migration window, no dual-hash period, no deprecation mechanism. When SHA-256 goes, all existing content is simultaneously unverifiable with new tooling.

**`generated_by` is called "informational" but IS included in `meta_hash`.** Section 6.3.7 says this field "does not affect provenance verification" — but Section 8 includes all fields except `meta_hash` and `signature`. So changing `generated_by` from `claude-code/4.0` to `claude-code/4.1` invalidates `meta_hash`. The spec's own language contradicts the math. This will cause implementers to treat it as a non-integrity field and corrupt their hashes.

## Logical Inconsistencies

**TV-15 is labeled "Informative" inside a normative appendix.** The appendix header says "Normative." TV-15 says "Informative." Which is it? If the test vector is informative, can implementations ignore it? The answer matters because TV-15 exists to justify the algorithm design decision. Move it to an informative appendix or change its status.

**`generated_by` is for "AI agents that generate content" but authors field guidance says "AI agents that generate content are tools, not authors."** Fine — but who records `generated_by` when an AI wrote 80% of the skill and a human wrote 20%? The spec doesn't say. And when content is updated across versions, should `generated_by` change if the v2 was human-written? No guidance.

**Section 10.2 version reset creates a silent ambiguity.** A fork at version 5 resets to version 1. A consumer comparing two content items by version would think the fork is older. The spec doesn't say that derived `version` doesn't inherit source versions, so consumers have to read the spec carefully to not misinterpret this. If version numbers don't convey progression across derivation boundaries, they could cause sorting/comparison bugs in registries.

**Composition is "implied" by list length, not stated.** Two `derived_from` entries = composition. But what if you adapted one source twice (iteratively, in different steps)? You'd have two entries with `adapt`, both pointing at versions of the same source. The spec would read this as composition, not serial adaptation. There's no way to represent "I adapted A to get B, then adapted B to get C" in a single `derived_from` list — you'd need a chain of `meta.yaml` files. This isn't wrong, but the spec doesn't say this.

## Underspecified Behaviors

**Unknown YAML fields: silent ignore or reject?** If a consumer encounters a `meta.yaml` with a field it doesn't recognize, should it reject? Warn? Silently proceed? The spec says nothing. In a versioned format, unknown fields are inevitable. JCS serialization of "all fields except meta_hash and signature" means unknown future fields get included in the meta_hash — which means a v1 consumer can't even verify meta_hash correctly for v2 content. This is a forward-compatibility hole.

**Sigstore verification requires network access, not mentioned.** Rekor lookup is required for Sigstore verification (Section 9.2). What happens in an air-gapped environment? What if Rekor is unreachable? The spec is silent. "Verification failure MUST result in rejection" — does a network timeout count as failure? This needs to be addressed, even if the answer is "use SSH signing for offline scenarios."

**OIDC identity matching is unspecified.** Section 9.2 says "Verify the OIDC identity in the certificate matches `signature.identity`." Exact string match? Prefix? Wildcard? `github.com/alice/code-review@refs/heads/main` — does `github.com/alice/code-review@refs/heads/*` match? This is the entire trust decision for signed content and it's left completely open.

**No definition of content directory boundaries in a monorepo.** Section 3 says a content directory contains "a content item's files." But if you have `/skills/code-review/SKILL.md` and `/skills/react-testing/SKILL.md` in the same repo, where does the content directory boundary go? The spec assumes content items are self-contained directories but never says how you determine that from a repo layout.

**The `source` field in `derived_from` "is NOT a machine-resolvable identifier."** Fine — but then what IS the authoritative lookup mechanism? Section 6.3.14 says `source_hash` is the "authoritative cross-reference anchor." But `source_hash` alone, without a resolvable location, means a consumer can verify the hash but can't retrieve the source to inspect it. The trust chain is verifiable in theory but not in practice without the `source` hint.

## Implementation Traps

**Windows + git autocrlf = broken verification.** Section 7.5 says "CRLF content produces a different hash than LF content" and "No CRLF normalization is performed." Section 5.1 says publishers MUST use LF. But a Windows developer who clones content with `core.autocrlf=true` will get CRLF files locally. Their consumer will recompute a different hash. Verification fails. The spec correctly defines the rule but doesn't warn about this sharp edge. It needs a prominent note or a RECOMMENDED `git` config for content repos.

**YAML integer `version: 1` serializes to JSON integer `1` — but what about `content_hash: "sha256:..."` with quotes?** In YAML, the `content_hash` value has explicit quotes. Strip the quotes and it's still a string. But an implementation that parses YAML and gets a string, then emits it as a JSON string — that's fine. However, `version: 3` is a YAML integer. When serialized to JCS it becomes `3` (integer). This is probably fine but the spec should explicitly say "YAML scalar types map to their JSON equivalents" to prevent a confused implementation from serializing version as `"3"` (string) instead of `3` (integer), which would invalidate meta_hash.

**The test vector hex for TV-13 PNG is wrapped across lines with an unclear concatenation instruction.** The note says "Concatenate all lines (removing whitespace)." But the hex is presented in a code block with visible newlines. An implementer copy-pasting this will get newline characters in their hex string if they're not careful. The hex should be presented as a single unbroken string, or the word "whitespace" should be "newlines and spaces."

**Section 7.3 step 2 says "Paths MUST NOT begin with ./ or end with /"** — but doesn't say they can't be empty strings. An edge case: what if the content directory path resolution produces an empty relative path? This can't happen with sane implementations, but it's an unhandled case in the spec.

**`authors` has no normalization requirement.** "Alice <alice@example.com>" and "alice <alice@example.com>" are different strings, produce different meta_hashes, and both are valid per the spec. No deduplication requirement. No case normalization. A tool that auto-generates author strings differently on different runs will produce non-reproducible meta_hashes.

## Smaller Issues

**Section 6.4 table says `authors` is required for "team" scope but optional for "local."** Why would local-use content need no authors? If I'm building a personal library of skills that I later promote to team scope, I now have to add authors retroactively and recompute meta_hash, breaking any local verification state. There's no migration path for scope promotion.

**The `published_at` MUST be "computed by tooling at publish time"** — but there's nothing stopping a tool from lying about this. It's not in the signature input (the signature covers hashes, not the raw field values), so a falsified `published_at` will still pass verification. At minimum, say it's a self-reported claim, consistent with Section 11.3.

**Appendix B verification status note ("MUST be confirmed by at least two independent implementations") is a process note embedded in normative test vectors.** That sentence describes a publication gate, not a test requirement. It belongs in a "Status of This Memo" section or the introduction, not in Appendix B.

**`type: mcp` with no guidance on credential handling.** MCP config files frequently contain tokens, API keys, or endpoint URLs. These get hashed and published with the content. The spec has a blanket "all files in the content directory are included, including hidden files" — meaning `.env` or credential files would be included in the hash and potentially redistributed. At minimum, warn that sensitive files should not be in content directories. Better: define a `.acpignore` mechanism.

**No `deprecated` or `yanked` signal.** If a published skill has a security flaw, there's no way to mark it as unsafe without controlling the registry. The spec has no field for "do not install this version" flags. This is a real ecosystem gap for anything meant to be distributed at scale.

**The `name` field in Section 6.3.3 says it's "human-readable" with no format constraints.** No max length, no forbidden characters, no slug requirements. A registry trying to build a namespace of `publisher/name` will have to deal with names containing slashes, newlines, null bytes, emoji. Worth adding basic constraints.

---

## The Single Biggest Issue

The spec says consumers MUST reject content where hash verification fails, and MUST reject when signature verification fails — but there's no guidance on what "reject" means in context. Reject from installation? Reject from loading into a running agent? Reject from display in a UI? "Reject" is doing a lot of work here. An agent that loads skills at startup and silently skips unsigned ones is technically conforming. An agent that hard-crashes on hash failure is also conforming. The behavioral contract is undefined.

---

This is a solid foundation — the hash binding mechanism (content_hash inside meta_hash) is smart, the JCS choice is correct, and the lineage model is clean. The sharp edges are mostly in edge cases and underspecified behaviors that will only surface when two independent implementations compare notes. Get those test vectors verified by a second implementation before calling this anything beyond draft.
