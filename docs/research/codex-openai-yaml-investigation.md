# Investigation: Codex `openai.yaml` Sidecar Format

> Research compiled 2026-03-22. Investigating the openai.yaml metadata sidecar for Codex CLI skills.

## Summary

The `agents/openai.yaml` file is a Codex-specific sidecar that sits alongside SKILL.md in a skill directory. It serves three purposes: UI presentation metadata (display name, icons, brand color), invocation policy (implicit vs explicit), and MCP tool dependency declarations. It is **not part of the Agent Skills specification** at agentskills.io — it's a platform-specific extension. No other provider (Claude Code, Cursor, Kiro, Copilot, Windsurf, etc.) has an equivalent sidecar file; they rely solely on SKILL.md frontmatter. The file is relatively simple (three top-level sections, ~10 leaf fields), auto-generated by Codex tooling, and primarily cosmetic except for the `dependencies.tools` section which has functional significance for MCP auto-wiring.

## Schema

The file uses three top-level sections:

```yaml
interface:
  display_name: "Title Case Skill Name"       # string, optional — UI-facing name
  short_description: "Brief summary"           # string, optional — UI-facing description
  icon_small: "./assets/small-logo.svg"        # path, optional — relative to skill dir
  icon_large: "./assets/large-logo.png"        # path, optional — relative to skill dir
  brand_color: "#3B82F6"                       # hex string, optional — UI theming
  default_prompt: "Prompt text for invocation" # string, optional — pre-filled prompt

policy:
  allow_implicit_invocation: false             # bool, default: true — when false,
                                               # skill only activates via explicit $skill

dependencies:
  tools:
    - type: "mcp"                              # string — tool protocol
      value: "openaiDeveloperDocs"             # string — tool identifier/name
      description: "OpenAI Docs MCP server"    # string — human-readable explanation
      transport: "streamable_http"             # string — connection method
      url: "https://developers.openai.com/mcp" # string — endpoint URL
```

### Field Details

| Field | Type | Required | Default | Purpose |
|-------|------|----------|---------|---------|
| `interface.display_name` | string | no | derived from skill name | Title Case name for UI skill lists and chips |
| `interface.short_description` | string | no | — | Brief summary for UI display |
| `interface.icon_small` | path | no | — | Small icon (SVG recommended, use `currentColor` fill) |
| `interface.icon_large` | path | no | — | Large icon (PNG or SVG) |
| `interface.brand_color` | hex | no | — | Hex color for UI theming |
| `interface.default_prompt` | string | no | — | Pre-filled prompt when invoking the skill |
| `policy.allow_implicit_invocation` | bool | no | `true` | Whether Codex can auto-trigger based on description match |
| `dependencies.tools[].type` | string | no | — | Protocol type (currently only `"mcp"` documented) |
| `dependencies.tools[].value` | string | no | — | Tool identifier that Codex checks for at invocation time |
| `dependencies.tools[].description` | string | no | — | Human-readable tool explanation |
| `dependencies.tools[].transport` | string | no | — | Connection method (e.g., `"streamable_http"`) |
| `dependencies.tools[].url` | string | no | — | Tool endpoint URL |

### Generation

The file is auto-generated by `scripts/generate_openai_yaml.py` or `scripts/init_skill.py` in the openai/skills repo. The `display_name` is derived from the skill directory name using Title Case conversion with special handling for acronyms (e.g., "gh-fix-ci" becomes "GH Fix CI", "github" becomes "GitHub"). The `short_description` and `default_prompt` are derived from SKILL.md content.

### Validation Constraints

- Skill name: regex `^[a-z0-9-]+$`, max 64 chars, no leading/trailing/consecutive hyphens
- Description: max 1024 chars, no `<` or `>` characters
- Icons should use `currentColor` fill for theme adaptation
- The file should be regenerated when SKILL.md changes (staleness check recommended)

## Real-World Examples

### Example 1: PR Babysitter (openai/codex repo)

From `.codex/skills/babysit-pr/agents/openai.yaml`:

```yaml
interface:
  display_name: "PR Babysitter"
  short_description: "Watch PR CI, reviews, and merge conflicts"
  default_prompt: "Babysit the current PR: monitor CI, reviewer comments, and merge-conflict
    status (prefer the watcher's --watch mode for live monitoring); fix valid issues,
    push updates, and rerun flaky failures up to 3 times..."
```

This skill uses only the three core `interface` fields — no icons, brand color, policy, or dependencies.

### Example 2: Plan (openai/skills repo, .system/plan)

The `.system/plan/` skill directory contains `SKILL.md` and `agents/openai.yaml`. It is a system skill (auto-installed) that operates entirely through prompt-based plan generation, requiring no scripts, references, or assets.

### Example 3: Documented Reference

From the official docs:

```yaml
interface:
  display_name: "Optional user-facing name"
  short_description: "Optional user-facing description"
  icon_small: "./assets/small-logo.svg"
  icon_large: "./assets/large-logo.png"
  brand_color: "#3B82F6"
  default_prompt: "Optional surrounding prompt to use the skill with"

policy:
  allow_implicit_invocation: false

dependencies:
  tools:
    - type: "mcp"
      value: "openaiDeveloperDocs"
      description: "OpenAI Docs MCP server"
      transport: "streamable_http"
      url: "https://developers.openai.com/mcp"
```

### Observed Patterns

In practice, most skills only use the three core `interface` fields (`display_name`, `short_description`, `default_prompt`). Icons, brand colors, and MCP dependencies are documented but appear to be used primarily by curated/partner skills rather than community-created ones. The `policy.allow_implicit_invocation` field appears rarely — most skills want to be implicitly discoverable.

## Cross-Provider Comparison

### Providers With No Equivalent Sidecar

| Provider | Skill Metadata Approach | UI Metadata |
|----------|------------------------|-------------|
| **Claude Code** | SKILL.md frontmatter only (name, description, plus CC-specific fields like allowed-tools, context, agent, model) | No UI metadata — CLI-only, no icons/colors |
| **Cursor** | SKILL.md frontmatter (name, description, license, compatibility, metadata, disable-model-invocation) | No UI metadata sidecar |
| **Kiro** | SKILL.md frontmatter (Agent Skills 5-field spec: name, description, license, compatibility, metadata) | No UI metadata sidecar |
| **Copilot CLI** | SKILL.md frontmatter (name, description, license, argument-hint, user-invocable, disable-model-invocation) | No UI metadata sidecar |
| **Windsurf** | SKILL.md frontmatter (name, description only) | No UI metadata sidecar |
| **OpenCode** | SKILL.md frontmatter (Agent Skills 5-field spec) | No UI metadata sidecar |
| **Amp, Cline, Roo Code** | SKILL.md frontmatter (name, description only) | No UI metadata sidecar |

### Key Differences

**Codex is the only provider with a sidecar metadata file.** This is because:

1. **Codex has a GUI app** (not just CLI) — it needs display metadata (icons, colors, display names) that CLI tools don't need.
2. **Codex has MCP auto-wiring** — the `dependencies.tools` section lets Codex automatically install and configure MCP servers when a skill is invoked.
3. **Codex separates agent-visible from UI-visible metadata** — SKILL.md frontmatter is for the AI agent's consumption (triggering decisions), while openai.yaml is for the UI layer.

### Related Patterns in Other Ecosystems

- **Cursor `.mdc` rules** have frontmatter (description, globs, alwaysApply) but no UI branding metadata.
- **GitHub Copilot** has `.github/copilot-instructions.md` and `*.instructions.md` with `applyTo` frontmatter, but no icons/branding.
- **The Agent Skills spec** (agentskills.io) defines the core SKILL.md format with 5 fields (name, description, license, compatibility, metadata) but explicitly does not define any sidecar format. The `metadata` map is the spec's extensibility mechanism — platforms could store UI metadata there instead of in a sidecar.

### Emerging Standards

There is no emerging cross-provider standard for UI metadata sidecar files. The openai.yaml pattern is Codex-specific. The closest thing to a cross-provider extensibility point is the `metadata` map in the Agent Skills spec, which any provider can use for custom key-value pairs.

## Recommendation

### Do Not Add to Canonical Format

The `openai.yaml` sidecar should **not** be added to syllago's canonical SkillMeta format. Here's why:

1. **It's Codex-only.** No other provider reads or writes this file. Adding it to the canonical format would mean carrying dead weight for 10+ providers to serve 1.

2. **Most fields are cosmetic.** `display_name`, `short_description`, `icon_small`, `icon_large`, `brand_color` are purely UI presentation for the Codex app. They don't affect skill behavior and can't be meaningfully converted to other providers (which are CLI-only).

3. **`default_prompt` is auto-generated.** It's derived from the SKILL.md body by tooling, not authored by hand. Including it in canonical format would be redundant with the skill body itself.

4. **`policy.allow_implicit_invocation` maps to existing fields.** Claude Code's `disable-model-invocation` serves the same purpose. The canonical format already handles this.

5. **`dependencies.tools` is the only functionally interesting part**, but it's MCP configuration — syllago already has a dedicated MCP content type with its own converter. Embedding MCP deps inside a skill sidecar would create a parallel path for MCP config that conflicts with the existing content model.

### Recommended Approach: Preserve Opaque on Codex Round-Trip

When importing a Codex skill that has an `agents/openai.yaml` file:

- **Store the file verbatim** in the skill's `.source/` directory alongside the original SKILL.md.
- **When rendering back to Codex**, restore the `agents/openai.yaml` file from `.source/` if present.
- **When rendering to other providers**, ignore it (no conversion needed — other providers don't use it).
- **Do not parse or represent its fields in SkillMeta.** Treat it as an opaque provider-specific artifact.

This gives lossless Codex-to-Codex round-trips without polluting the canonical format or requiring converter changes for every openai.yaml schema update.

### Future Considerations

If other providers adopt similar sidecar files (plausible if more tools get GUI apps), revisit this decision. The right trigger would be 2+ providers having functionally equivalent sidecar formats, at which point a canonical `SkillUI` metadata struct might make sense. Until then, opaque preservation is the right call.
