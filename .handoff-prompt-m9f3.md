# Fix: syllago add only copies primary content file, drops supporting files (syllago-m9f3)

## Bug

When `syllago add` imports a skill (or other content type) that has supporting files in subdirectories, only the primary file (e.g., SKILL.md) is copied to the library. All supporting files (workflows/, scripts/, references/) are silently dropped.

This breaks the entire add → library → install → publish pipeline for multi-file content.

## Reproduction

```bash
# Source skill has 10 files:
# skills/atlassian/SKILL.md
# skills/atlassian/workflows/jira/create-issue.md
# skills/atlassian/workflows/jira/search-issues.md
# skills/atlassian/workflows/jira/update-issue.md
# skills/atlassian/workflows/jira/view-issue.md
# skills/atlassian/workflows/confluence/create-page.md
# skills/atlassian/workflows/confluence/search-pages.md
# skills/atlassian/workflows/confluence/update-page.md
# skills/atlassian/workflows/confluence/view-page.md
# skills/atlassian/workflows/search-atlassian.md

syllago add skills/atlassian --from claude-code

# Library result: only SKILL.md — workflows/ directory missing
ls ~/.syllago/content/skills/atlassian/
# SKILL.md    (that's it)
```

## Expected behavior

All files in the content item's directory tree should be copied to the library, preserving the directory structure.

## Where to look

The add pipeline has these key files:

1. **`cli/internal/add/add.go`** — `DiscoverFromProvider()` discovers items, `AddItems()` writes them to library. This is where the single-file copy likely happens.
2. **`cli/cmd/syllago/add_cmd.go`** — CLI wiring that calls into `internal/add`. The `converterAdapter` only canonicalizes the primary file.
3. **`cli/internal/catalog/`** — `ContentItem.Files` field already tracks multiple files per item. The scanner knows about multi-file items. The add pipeline just doesn't use this.

The discovery step likely finds the directory but the write step only copies the primary content file (SKILL.md, AGENT.md, etc.) rather than the full directory tree.

## Fix approach

1. Read `cli/internal/add/add.go` to understand how `AddItems()` currently works
2. Find where it copies the single file instead of the directory
3. Change it to copy the entire item directory recursively (preserving structure)
4. Make sure `.syllago.yaml` metadata still gets written at the item root
5. Update `syllago publish` if it also does single-file copy (check `cli/cmd/syllago/publish_cmd.go`)
6. Add a test with a multi-file skill fixture
7. Run `make build && make test`
