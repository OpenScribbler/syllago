## v0.10.3

### Highlights

**Cross-provider loadout apply restored.** A single-source loadout (e.g.
`provider: claude-code`) applied to a different target via `--to <other>`
read as `unresolved references: hooks: ... not found` since v0.10.0. The
multi-provider loadout work in v0.10.0 narrowed reference resolution to the
target slug only — correct for true multi-provider loadouts, but it broke
the cross-provider conversion workflow where syllago is meant to read the
source provider's content and convert it to the target's paths/format on
install.

Resolution now tries the target slug first, then falls back to the
manifest's declared source providers (`manifest.Provider`, then each
`manifest.Providers` entry). When both flavors exist, the target still
wins; the existing multi-provider behavior is unchanged.

---

### Bug Fixes

- **`syllago loadout apply <name> --to <other>` for single-source loadouts.**
  Reference resolution falls back to the manifest's source provider when the
  target has no flavored copy of a hook, rule, or command. Pinned by
  `TestResolve_CrossProviderFallback`; multi-provider precedence pinned by
  `TestResolve_TargetWinsOverFallback`.
- **`templates/hooks/hook.json` parse error.** The shipped template used
  the legacy flat hook shape (top-level `event`/`matcher` and a flat
  `hooks: [{type, command}]` array). `Manifest.ParseManifest` rejects any
  manifest missing `spec: "hooks/0.1"`, so anyone scaffolding a hook from
  the template hit `missing spec field (expected "hooks/0.1")`. Template
  now matches the canonical shape: top-level `spec`, nested `handler{}`.

### Internal

- Smoke fixture `tests/smoke/fixtures/hooks/claude-code/smoke-hook/hook.json`
  conformed to canonical `hooks/0.1`. Same legacy-shape parse error had been
  failing the Claude Code and Gemini provider smoke workflows since v0.10.0.
- Smoke test `Preview mode` no longer passes the removed `--preview` flag;
  running `syllago loadout apply <name>` with no mode flag is the canonical
  preview invocation.
