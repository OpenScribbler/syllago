## v0.7.1

### Highlights

**Complete provider capability data** — Every supported content type (skills, rules, hooks, MCP, agents, commands) now has structured capability data for every provider that supports it. Cross-provider conversion and the provider information shown in `syllago info` use this data to be accurate end-to-end.

**Capabilities organized by content type** — New `docs/provider-capabilities/by-content-type/` directory groups all provider data by content type (`skills.yaml`, `rules.yaml`, `hooks.yaml`, `mcp.yaml`, `agents.yaml`, `commands.yaml`), making it easier to compare what each provider supports for a given content type.

---

### New Features

#### Provider Capability Coverage

For each content type, every provider that supports it now has machine-readable capability data published in `docs/provider-capabilities/`:

- **Skills** — All 14 providers covered.
- **Rules** — 10 providers covered, with activation modes, frontmatter fields, glob support, hierarchical loading, and cross-provider file recognition (`AGENTS.md`, `CLAUDE.md`, etc.) where applicable.
- **Hooks** — 9 providers covered, with handler types, matcher patterns, decision control, scopes, and JSON I/O protocol details where applicable.
- **MCP** — 9 providers covered, with transport types, OAuth support, environment-variable expansion, tool filtering, marketplace, resource referencing, and enterprise management where applicable.
- **Agents** — 8 providers covered, with definition format, tool restrictions, invocation patterns, scopes, model selection, per-agent MCP, and subagent spawning where applicable.
- **Commands** — 4 providers covered, with argument substitution and built-in commands where applicable.

For providers that document a feature differently or do not support it, the capability data records that explicitly so the docs and conversion logic can act on it.

#### Capmon Tooling
- **`syllago capmon generate`** — Produces the per-content-type aggregate files in `docs/provider-capabilities/by-content-type/` from the per-provider YAML files. Run after editing or regenerating any provider's capability data.
- **`syllago capmon seed --provider <slug>`** — Re-seed a single provider's capability YAML from cached source data. Useful when iterating on one provider at a time.
- **Nested sub-capability output** — The seed writer correctly nests dotted capability paths under `capabilities` instead of flat-emitting them, so files like `cross_provider_recognition.agents_md` now produce structured sub-blocks.

### Improvements

- **Codex skills extraction** — Extracts the right set of struct definitions (no more accidental matches on similarly-prefixed types).
- **TypeScript and Rust field extraction** — Interface fields (TypeScript) and struct fields (Rust) are now extracted alongside the type names, giving capability data more detail for providers that publish typed source.
- **Hook spec generator** — Skips capability YAML files with empty slugs instead of failing.
- **Test isolation** — `findProjectRoot` now stops at a configurable boundary in tests, preventing nested-fixture tests from walking into the real repo root.
- **Lint cleanup** — 27 outstanding lint findings resolved; CI is clean.
- **Dependencies** — `golang.org/x/net` promoted to a direct dependency in `go.mod` to match actual usage.

### Bug Fixes
- Fixed the seed writer flat-emitting nested sub-capabilities instead of producing structured blocks.
- Fixed the hook spec generator failing on empty-slug capability YAML files.
- Fixed `findProjectRoot` walking past test fixtures into the real repo root in nested-git-init test cases.
- Fixed an unused constant that was failing the lint gate.
