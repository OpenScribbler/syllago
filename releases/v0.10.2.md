## v0.10.2

### Highlights

**Provider detection rewritten to fix false positives.** Syllago's "installed"
detection used to ask "does the provider's config directory exist?" — but
because syllago itself writes content into those same directories
(`~/.claude/skills/`, `~/.cursor/skills/`, etc.), installing syllago content
made syllago report the provider as installed even when it wasn't. v0.10.2
replaces the directory check with per-provider host-tool signals: binary on
PATH, app-only marker files (e.g. `~/.claude.json`, `~/.codex/auth.json`),
VS Code extension presence (`~/.vscode/extensions/<id>-*/`), and `gh`
extension queries. Detection is now advisory — it never blocks an explicit
`--to <provider>`, and `Detect()` carries a doc comment saying so.

Every provider gained per-provider regression tests asserting that a home
directory containing only syllago-managed subdirectories does not produce a
false-positive detection.

---

### Bug Fixes

- **Provider detection across all 15 providers.** Replaced "is config dir
  present?" with heterogeneous signals:
  - **Standalone CLIs** (Claude Code, Codex, Gemini, OpenCode, Crush, Amp, Pi,
    Factory Droid): binary on PATH; Claude Code and Codex also accept their
    app-only marker files.
  - **Electron IDEs** (Cursor, Windsurf, Kiro, Zed): binary on PATH or
    OS-specific app data directory (`~/.config/<App>` on Linux,
    `~/Library/Application Support/<App>` on macOS). Zed uses
    `~/.config/zed/settings.json` rather than the bare config dir.
  - **VS Code extensions** (Cline, Roo Code): glob
    `~/.vscode/extensions/<id>-*/`.
  - **`gh` extension** (Copilot CLI): `gh extension list`.
- **Kiro detection on Linux.** Kiro's installer ships `kiro-cli`, not a binary
  named `kiro`, and writes its sqlite store to `~/.local/share/kiro-cli/`.
  Detection now matches the actual installer layout. The legacy `kiro` binary
  check is kept as a defensive fallback for future renames.
- **`capmon-check` GitHub Actions workflow.** The workflow runs from repo
  root, but the command's default `--providers-json` looked in the cwd while
  the file lives at `cli/providers.json`. The path is now passed explicitly
  to both invocations.
- **`capmon` source manifest validator.** Two manifests
  (`gemini-cli/agents`, `opencode/skills`) document content types implemented
  purely via cross-provider convention with no upstream URL to monitor — the
  validator was rejecting them as "no source URIs and not marked as
  unsupported." A new `convention:` field on `ContentTypeSource` names the
  cross-provider pattern (`cross-provider-agents-md`,
  `cross-provider-skill-md`) and satisfies the URI requirement; unknown
  conventions still fail with a descriptive error.
- **CLI privacy wording.** `publish`/`published` was inconsistent with the
  public `share` verb across privacy gates, error messages, comments, and
  tests. All user-facing strings now read `share`/`shared`.

### Build & Infrastructure

- Added `CODEOWNERS` covering security-sensitive paths (workflows, installer,
  sandbox, signing, updater, registry, `SECURITY.md`).

### Internal

- Removed the dead `dirHasUnmanagedEntry` helper (no remaining callers after
  every provider migrated to the new detection signals).
- Documented `Provider.Detect` as advisory — heuristic only, never a
  correctness gate. Explicit `--to` always works regardless of detection.
- Added five small detection helpers (`binaryOnPath`, `fileExists`,
  `dirExists`, `vscodeExtensionInstalled`, `ghExtensionInstalled`) reused by
  every `Provider.Detect` implementation. The `gh` helper uses package-level
  overrides so it is testable without requiring `gh` on the host.
