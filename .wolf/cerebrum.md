## Key Learnings

- **Project:** syllago
- **Description:** <div align="center">
- **Release pipeline is CI-driven.** `.github/workflows/release.yml` fires on `v*` tag push and does the full pipeline itself: builds six binaries (linux/darwin/windows × amd64/arm64), generates `commands.json`/`providers.json`/`telemetry.json`/`capabilities.json`, produces an SPDX SBOM via syft, signs checksums with cosign, creates the GitHub Release with all artifacts attached, and updates the Homebrew tap. Do NOT run `gh release create` manually — the `$PAI_DIR/hooks/release-guard.sh` hook will block it, and duplicating the work conflicts with the CI run. Flow: prep commit → push tag → wait for CI. The `release` skill's Step 8 still says to run `gh release create` manually; that guidance is stale for this repo.
- **Release workflow does not auto-detect pre-release tags.** `.github/workflows/release.yml` calls `gh release create` without `--prerelease`, so tags like `v0.7.0-beta.1` come out as stable releases. Workaround: `gh release edit <tag> --prerelease` after the run completes. Follow-up: teach the workflow to detect pre-release suffixes (`-alpha`, `-beta`, `-rc`, `-pre`) and pass the flag automatically.
- **`gh api /repos/…/actions/runs` paginates silently.** Default page size is 30, max is 100. A single call never reveals the real population — a workflow with 190 runs will return only the first 30 and give no hint more exist. Always check `.total_count` first (use `per_page=1` for a cheap probe), then paginate explicitly. For duration math across all runs, loop `page=1..N` until results are exhausted. Applies to `gh api /repos/{owner}/{repo}/actions/workflows/{id}/runs` too.
- **Actions billing attribution gotcha.** GitHub's billing UI uses the workflow's `name:` field, not the filename. CodeQL's default name is "CodeQL" and it's categorized under "Code scanning" in some views — easy to confuse with a Claude review workflow when both contain "code" in the name. When investigating cost, map filename → `name:` → billing label before drawing conclusions.

- **[2026-04-15] Internal labels in release notes.** Copied `D1/D13/D14-delta/D16` straight from a commit message into the 0.7.0-beta.1 release notes. The user called them "garbage" — they mean nothing to a reader who doesn't have the bead tracker open. Fix: describe what the user can now do (e.g., "Provider detail view: `syllago info providers <slug>`") instead of naming the work item.
- **[2026-04-15] Claimed local `go build` success = release-ready.** CleanStale in `cli/internal/sandbox/staging.go` used `syscall.Stat_t` and `os.Getuid` inline. This compiles fine on Linux but the Windows cross-compile in the release workflow failed with "undefined: syscall.Stat_t". For any release that ships Windows binaries, assume local `go build` is insufficient signal — either cross-compile for each target (`CGO_ENABLED=0 GOOS=windows GOARCH=amd64 go build ./cmd/syllago`) before tagging, or accept that the release workflow is the real validator and be ready to fix-and-retag.
- **[2026-04-17] When repo goes public, re-add `push: main` to scorecard.yml.** Trigger was removed while private because OSSF Scorecard's `publish_results: true` is a no-op on private repos — the SARIF gets uploaded to code-scanning but nothing reaches the public OpenSSF database. After the public flip, restore the `push: main` trigger so Scorecard updates the badge on every commit (current weekly cron becomes insufficient signal for a public-facing security score).
- **[2026-04-22] Never call a lint/test failure "pre-existing."** When a pre-push lint hook blocks, my push triggered it — even if the broken code was written in a prior session. The golden rule: all builds and tests are green upstream; if something fails, investigate and fix it. Never dismiss a failure with "pre-existing" or "not my change." The lint fix is always the next step. **Rule:** if lint blocks a push, diagnose which commit introduced the failure (via `git log` + `git blame`), fix it, amend or follow-up commit, and push cleanly.
- **[2026-04-22] `applyPrecedence` dedup key must include provider for all non-universal types.** Prior to 814bfa7, the dedup key for Rules/Hooks/Commands was `(name, type, "")` — no provider. So `content/rules/claude-code/concise-comments` and `content/rules/gemini-cli/concise-comments` both mapped to the same key, and only the first-scanned provider survived in `cat.Items`. The others went to `cat.Overridden` invisibly. Fix: for `!item.Type.IsUniversal()`, set `k.provider = item.Provider`. Universal types (Skills, Agents, MCP) keep empty provider since they are provider-agnostic. **Rule:** any catalog change that adds same-named content for multiple providers must re-verify `precedence.go` uses `!item.Type.IsUniversal()` scoping.
- **[2026-04-22] `Resolve()` must accept an explicit providerSlug, not read from manifest.Provider.** When a loadout uses `providers: [...]` (multi-provider), `manifest.Provider` is empty. If `Resolve` reads `manifest.Provider` to find provider-scoped catalog items, it passes `""` to `findItem`, which fails to match any provider-specific rule. Fix: change signature to `Resolve(manifest, cat, providerSlug string)` and have the caller (apply loop) pass the current target provider's slug. **Rule:** any function that filters catalog items by provider must take the provider slug as an explicit parameter, never derive it from the manifest.
- **[2026-04-20] Don't recommend architectural changes to syllago without reading `cli/internal/{loadout,installer,catalog,snapshot}/` first.** Wrote a research doc comparing MO2 to syllago and recommended "split install into manifest + reconcile with snapshot rollback" — which `loadout/apply.go:52` already does (Resolve → Validate → Preview → Snapshot → apply, with snapshot.Restore on failure). User had to ask for a second pass grounded in the actual code. **Rule:** before proposing any apply/install/conflict/catalog change, grep for the feature in the existing packages. `installer/orphans.go`, `catalog/precedence.go` (with `Overridden`), `snapshot/` atomic rollback, and `loadout/preview.go` cover more than their package names suggest.
- **[2026-04-22] Hook event names have two forms — canonical (`before_tool_execute`) and provider-native (`PreToolUse`) — and every site that uses an event as a JSON key MUST translate.** Library `hook.json` stores canonical names (produced by `SplitSettingsHooks` → `ReverseTranslateHookEvent` at add-time). Provider `settings.json` is read by the provider itself, so it MUST use native keys. `converter.IsValidHookEvent` accepts BOTH forms, so validation will not catch the mismatch — you'll silently inject hooks under keys the provider never reads. `converter.TranslateHookEvent(canonical, providerSlug) (native, ok)` returns `ok=false` for already-native names, so it's safe to call unconditionally. **Rule:** `installHook`, `uninstallHook`, `checkHookStatus`, and any future code that uses `event` as an `sjson` path key must translate canonical → native first. Covered in `installer/hooks.go` as of 2026-04-22 fix.
- **[2026-04-23] TUI add wizard `discoveredItems` has a layout invariant: `[actionable...] + [installed...]`. Any merge that adds StatusNew items MUST insert them in the actionable block — never append past `installed`.** `visibleDiscoveryItems()` returns `discoveredItems[:actionableCount]` when `!showInstalled && installedCount>0`, so `discoveryList.selected` is sized to `actionableCount`. Appending merged items past the installed block put their indices out of range of `selected[]`, causing `index out of range [N] with length N-mergedCount` panic from the post-merge selection loop. The fix: capture `preMergeActionableCount`/`preMergeInstalledCount` once in `handleDiscoveryDone`, rebuild as `[origActionable + merged, origInstalled]` on merge, update `actionableCount`. Also: the explicit selection loop was redundant — `buildDiscoveryList` already pre-selects StatusNew items. **Rule:** when changing `discoveredItems` layout, re-verify the invariant at every call site of `visibleDiscoveryItems()` and `buildDiscoveryList()`; pre-merge baselines are needed for idempotent Back→Triage→Next flows.
- **[2026-04-23] Hook drill-in must show the canonical flat hook.json + referenced scripts — never the settings.json fragment with `// Event:` comments.** Old `extractJSONSection` (add_wizard.go) produced invalid JSON for provider-sourced hooks in the wizard review drill-in and omitted bundled scripts entirely. Library + Content > Hooks already showed the right thing because `writeHookToLibrary` materializes `hook.json` + bundled scripts on disk and `catalog.collectFiles` walks the dir. The wizard was the odd one out because it previews BEFORE write. Fix: `buildHookPreviewFiles` synthesizes a virtual tree `[hook.json, <scripts...>]` from `item.hookData` + `item.hookSourceDir`; `readHookPreviewContent` returns `json.MarshalIndent(hookData)` for hook.json and reads only script files that are actually referenced by a hook command (guards against reading random files from the provider's settings dir). Wired into `enterReviewDrillIn` and `loadDrillInFile`. **Rule:** any TUI surface that renders a hook MUST show `hook.json` as the flat single-hook object plus the scripts it references — never settings.json fragments, never "// Event:" pseudo-JSON. Use the virtual tree helpers even if the hook isn't on disk yet.
- **[2026-04-22] TUI "Add from provider" for hooks/MCP cannot route through the generic `add.AddItems` copy-from-path flow.** `add.writeItem` calls `os.ReadFile(item.Path)` and writes the result to `hook.json` / `mcp.json`. For file-based types (rules, skills) `Path` points to a real file. For hooks, `Path` is a *settings.json* that contains many hooks — copying it wholesale produces a broken library file. Pattern: extract the structured data at discovery time (`converter.SplitSettingsHooks` for hooks, equivalent for MCP), stash it on the discovery item, and branch in `addSingleItem` to serialize structured data to `hook.json` via `json.MarshalIndent` + bundle scripts + write `.syllago.yaml`. Mirror the CLI's `addHooksFromLocation` in `cli/cmd/syllago/add_cmd.go`. **Rule:** any provider-sourced content type where the on-disk representation is a multi-item container (settings.json, mcp.json) needs a type-specific write path in the TUI, not the generic copy flow.

- **[2026-04-23] Never call a test failure "pre-existing" or "unrelated" when the upstream branch is green.** Working-tree had 111 unstaged deletions of `content/hooks/**` which made `TestScanRealRepo` fail. Dismissed it as pre-existing local state; the stop hook called it out: *"NOTHING IS PRE-EXISTING. All builds and tests are green upstream. If something fails, YOUR work caused it."* Fix was `git restore content/` to bring the tracked files back. **Rule:** if any test fails, diagnose + fix — no exceptions. Restoring tracked files via `git restore <path>` is the default remedy for unstaged deletions that arrived during the session. Don't debate authorship; fix it.

- **[2026-04-23] `syllago self-update` and `syllago registry approve` are phantom commands that don't exist.** The real binary updater is `syllago update`. The `registry approve` command was planned but never implemented — recovery for a changed signing profile (exit 11) is `syllago registry remove <name>` then `syllago registry add <url>`. Recovery for an unpinned profile (CodeIdentityUnpinned warning) is `syllago registry sync <name>`. **Rule:** before writing any user-facing Suggestion string referencing a CLI command, grep `Use:` in `cmd/syllago/` to confirm the command actually exists.

## Decision Log

<!-- Significant technical decisions with rationale. Why X was chosen over Y. -->

- **[2026-04-15] Force-moved `v0.7.0-beta.1` tag instead of burning version.** First tag push pointed at a commit that failed CI (Windows build). Options were: (a) burn the tag and use `v0.7.0-beta.2`, (b) force-update the tag to the fix commit. Chose (b): the tag had been public ~5 minutes, CI failed before creating the GitHub Release, no Homebrew push happened, no one could have pulled it. Force-moving is acceptable for a tag that never corresponded to a real release. Would not have done this if a release had already been created or if the tag had been public longer.
- **[2026-04-15] Windows `isOwnedByCurrentUser` returns true (no-op) rather than implementing Windows ACL checks.** The function gates stale-staging-dir cleanup on ownership to avoid deleting other users' dirs in shared `/tmp`. On Windows, temp paths are per-user (`%LOCALAPPDATA%\Temp`, `C:\Users\<user>\AppData\...`) so cross-user contention doesn't arise the same way. Implementing real Windows ACL introspection (via `golang.org/x/sys/windows`) would add a dependency and complexity for no observable benefit. Revisit only if a real Windows sandbox use case emerges.
